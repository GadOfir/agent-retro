Features

After a long session, triage what went wrong

Your agent just worked for hours. Some steps succeeded. Some didn't. The retro captures what broke — organized so you can see patterns across sessions.

Per model

Which model was running when it broke? Claude 4? GPT-4o? o3-pro? Log it. After a few sessions you'll see which models struggle with which step types.

"Claude 4 works great on refactors, struggles on test generation"
Per harness

Which harness was driving it? Claude Code? Codex? Cursor? Different harnesses have different context windows, different tool access. The same model behaves differently across them.

"Same model, Codex got it right, Claude Code missed"
Per task step

What was the agent trying to do when it broke? Testing? Deployment? Architecture decision? Code review? Each step type has its own failure patterns.

"Testing stage fails: agent writes tests that pass but don't test the right thing"
Agent self-reports

You don't need to reconstruct everything from logs. The agent fills out the retro itself — what went wrong, at which step, with which model, in which harness.

"Step 3 broke. Model: Claude 4. Harness: Claude Code. Here's why."
Classification baked in

Each entry gets a class: was the rule missing (CX-CANDIDATE)? Was it a bad assumption (AGENT-NOTE)? Confusing code (CODE-SMELL)? The class prescribes the fix.

CX-CANDIDATE → file a gap. AGENT-NOTE → track it. CODE-SMELL → log it.
Patterns over sessions

One retro is noise. Five retros is a pattern. "Every time we use model X with harness Y on testing, it goes off the rails. Let's stop doing that."

NOTES.md accumulates, dedupes, escalates on recurrence

Workflow

End-of-session loop

1

Agent finishes a session

Your agent just spent hours on a task — coding, testing, deploying. Some steps worked. Some didn't. You're done coding. Now you capture the learning.

2

Agent self-reports what went wrong

Run the retro skill. The agent fills out: which step broke, which model was running, which harness was driving it, what happened. Evidence-based — not from memory, from the session artifacts.

3

You triage — model × harness × step

Entries are organized three ways. Which models consistently struggle? Which harnesses produce different behavior? Which task steps are hotspots? The triage table surfaces patterns.

4

Improve the lifecycle

Find the common fix. Update contracts. Refine model routing. Adjust harness config. The same pattern won't repeat. Next session starts from what the last one learned.

Why

Every long session is data you're throwing away

The problem

Your agent just worked for hours. Three steps succeeded, two failed, one took a detour. You ship the code but lose the learning. Next session, you route the same model to the same broken step. The lifecycle doesn't improve because nothing was captured.

The fix

End every long session with a retro. The agent self-reports: what broke, which model, which harness, which step type. You triage into a table. Patterns emerge across sessions. "Every time Claude 4 runs testing in Codex, it misses edge cases." Fix it once — route a different model there next time.

The edge

One retro teaches you about one session. Accumulate five and you have a map: this model works for that step type, this harness gives better results there, this step always needs a human review. Your agent lifecycle gets measurably better. Not because the AI improved — because your routing did.

The core

Triage across three dimensions

Model × Harness × Step

After the agent self-reports, you sort every entry into a triage table. Model: which AI was running? Harness: which tool was driving it? Step: what was it trying to do when it broke? Each cell in this grid tells a story. One cell with five entries is a hotspot. One model dominating one step type is a routing problem.

The classification (CX-CANDIDATE, VERIFICATION-GAP, AGENT-NOTE, OPERATOR-NOTE, CODE-SMELL) tells you what kind of fix each entry needs. But the triage tells you where to aim it. "Testing steps keep failing under Claude 4 in Codex" is actionable. "Agent was bad" is not.

This is the difference between retro that feels like therapy and retro that actually improves your next session. Sort first. Classify second. Fix once.

Use cases

When to reach for this

Long coding session just finished

Agent worked for 2+ hours. Multiple steps: scaffolding, logic, tests, deploy. Some steps were smooth, others were a fight. Run the retro and log what happened per step.

Testing stage keeps failing

Agent writes tests that pass but don't actually test the right thing. Or tests fail for reasons the agent can't diagnose. Run retro to capture: which model, which harness, what confused it.

Agent got stuck in a confusion loop

Agent tried approach A, reverted, tried B, reverted, tried A again. Two hours wasted. Retro captures what it was confused about — was it the code? The architecture? Missing context?

Session ran across multiple models

You routed different models to different steps. Retro logs which model handled which step and how each performed. Next session, your routing is data-driven.

Weekly review of agent performance

Accumulate retro entries over a week. Sort by model, harness, step type. "Claude 4 handles refactors perfectly. GPT-4o keeps inventing APIs that don't exist. o3-pro is best on testing." Your lifecycle is now evidence-based.

Onboarding a new harness

You just added Codex to your workflow. Run retros for the first week to see how it compares to Claude Code on each step type. Data tells you where each harness belongs.

FAQ

Questions you'll have after a session

My agent just finished a 4-hour session. What do I do?

Run the retro skill. Tell your agent: "run a retro on this session." The agent self-reports: which steps broke, which model was running, which harness, what went wrong. You triage the results and look for patterns.

Should I run this after every session or only bad ones?

After every long session — even the smooth ones. Sessions that went well still have edges: a step that was slower than expected, a model that needed extra prompting, a harness that lost context. Patterns emerge from volume, not just from failures.

How do I know which model to use for which step?

Run retros for a week. Sort by model × step type. The data tells you. "Claude 4 handles refactors and deploys cleanly. O3-pro is better at test generation. GPT-4o struggles with architecture decisions." Your routing becomes evidence-based, not gut-feel.

What's the difference between a model issue and a harness issue?

Same model, different harness = harness issue. Claude 4 in Claude Code vs Claude 4 in Codex — they have different context windows, different tool access. The retro captures which harness produces better results for each step type.
