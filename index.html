Features

What Agent Retro gives you

Not a post-mortem. A diagnostic framework that classifies every failure mode and prescribes the fix.

CX-CANDIDATE

Missing or vague rule that would have prevented the failure. File a gap and update the contract. Fix the system, not the symptom.

"The docs were silent on this" → CX gap
VERIFICATION-GAP

Agent claimed success but proof was below claim level. A fix shipped without evidence it actually worked. Flag the overstated hand-back.

"Seems fine" is not a test.
AGENT-NOTE

Bad call despite available info — wrong assumption, ignored contract, over-engineered. Append to NOTES.md. Second occurrence auto-escalates.

NOTES.md → deduped → CX
OPERATOR-NOTE

Ambiguous instructions or missing clarifying questions. Not a file — report inline. Improves operator-agent communication.

"You should have asked"
CODE-SMELL

Confusing code — misleading names, dead code that looks alive. One line + pointer in SMELLS.md. No essays. Not runtime state.

SMELLS.md → one line per smell
Protocol

How it works

1

Reconstruct from evidence

Git log, hand-back tables, RESULT.md, CX gaps filed mid-session, dead-approaches register. Every claim re-derived from an artifact — never from narrative memory.

2

Classify each issue

Five classes: CX-CANDIDATE, VERIFICATION-GAP, AGENT-NOTE, OPERATOR-NOTE, CODE-SMELL. Each has a definition and a prescribed action.

3

Output mandatory sections

Structured output summarizing findings, root causes, and fixes per class.

4

Apply the fixes

File CX gaps, update contracts, append to NOTES.md (deduped), update dead-approaches register, append to SMELLS.md. Observe first — fix the system, not the blame.

Why

A retro that retrofits your system

The problem

AI agents make bad calls. They ship fixes without evidence. They repeat mistakes. They make assumptions that were never written down. Without a structured retro, every session starts from zero and the same failures recur.

The fix

Evidence-based classification. Every issue lands in one of five classes. Each class has a prescribed action. CX gaps update the contract. AGENT-NOTEs get deduped and escalate on recurrence. The system improves over time because the retro fixes the guide rails, not the person.

The edge

No blame. The agent didn't fail — the system failed to guide it. Retro fixes the system. When the retro blames the agent, the agent learns nothing. When it fixes the contracts, every session gets better.

The core

Five classes. One root cause each.

The classification matrix

Every failure mode maps to exactly one class. CX-CANDIDATE means a rule was missing — file a gap and the system learns. VERIFICATION-GAP means proof didn't match the claim — the testing contract needs hardening. AGENT-NOTE means bad judgment despite available info — track it and escalate on repeat.

OPERATOR-NOTE means the instructions were ambiguous — report inline, no file. CODE-SMELL means the codebase was confusing — one line in SMELLS.md, no essay, never fix during retro.

This taxonomy prevents three common retro failures: vague blame ("the agent was bad"), unfixable action items ("be more careful"), and mixing cleanup with diagnosis. Classify first. Fix second.

Use cases

When to reach for this

Post-deployment review

A deploy shipped with issues. Run the retro to classify what went wrong and file CX gaps so the next deploy is safer.

Recurring agent mistake

Same bad pattern appears twice. The retro detects recurrence and auto-escalates AGENT-NOTE to CX-CANDIDATE.

Agent handoff

Agent A worked on a session. Agent B picks up the retro. The artifact trail gives B everything needed without repeating A's mistakes.

Pre-audit cleanup

Before running a full contract audit, run a retro to catch known failure modes. Reduces audit noise.

FAQ

Questions you'll ask after a rough session

How is this different from a post-mortem?

A post-mortem assigns blame. Agent Retro classifies failure modes and fixes the system. CX gaps update contracts. NOTES.md dedupes lessons. The system gets better over time.

When should I run this?

When the agent shipped something but the journey was painful. After a deploy that required reverts. When you see a pattern repeat. NOT for product gaps — those go to your product process.

What if the same issue keeps happening?

That is exactly what this skill catches. Second occurrence of an AGENT-NOTE pattern auto-escalates to CX-CANDIDATE. A recurring mistake is a missing rule, not bad luck.

Can I skip the evidence reconstruction?

No. A retro built on memory alone re-creates the fabricated-timeline risk. Every claim must be re-derived from an artifact: git log, hand-back tables, RESULT.md, CX gaps. Commits alone are not enough.
